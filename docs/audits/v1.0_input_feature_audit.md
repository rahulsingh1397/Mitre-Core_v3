**Date:** 2026-05-24
**Status:** Complete (Stage A + Stage B done; verdict MATERIAL)
**Trigger:** Exp 3 (15-dim features) pre-flight identified candidate leakage in proposed new features; deeper audit found the leakage was already present in the v1.0 baseline.
**Authored as part of:** Part X.A.IV of the working plan

## TL;DR

The V3 baseline (`network_v9_v3` at `alert_feature_dim=6`) feeds **two label-derived columns as inputs at training and inference**: `tactics` (from `df["tactic"]`) and `alert_types` (binary collapse of `df["alert_type"]`). Both are evaluation labels. A shuffle test on all v1.0 + v1.1 frozen baselines produced these clean ARIs:

| Dataset | Reported | Clean | Δ | Verdict |
|---|---|---|---|---|
| NSL-KDD v1.0 | 0.602 | 0.093 | −0.509 | INVALIDATED |
| UNSW-NB15 v1.0 | 0.564 | 0.289 | −0.275 | INVALIDATED |
| TON-IoT v1.0 | 0.423 | 0.350 | −0.073 | MIXED (superseded by v1.1) |
| **TON-IoT v1.1 (GMM+BIC)** | 0.604 | 0.593 | **−0.011** | **GENUINE** |
| CICIDS2017 v1.0 | 0.177 | 0.186 | +0.009 | GENUINE (honest loser) |
| SQTK_SIEM v1.0 | 0.355 | 0.188 | −0.167 | INVALIDATED |
| SQTK_SIEM v1.1 (pca=11) | 0.461 | 0.162 | −0.299 | INVALIDATED + REGRESSES |
| DARPA OpTC v1.0 | 0.999 binary | −0.020 | −1.019 | CATASTROPHIC |

**Two findings survive the audit:** TON-IoT v1.1 GMM+BIC (genuine improvement) and CICIDS2017 v1.0 (honest loser). Everything else is an upper bound or invalidated.

## Evidence

### Source-code citations

**File:** `hgnn/hgnn_correlation.py` 

```python
# Line 752 — `tactics` feature pulls from the tactic LABEL column
tactics = pd.Categorical(df["tactic"]).codes if "tactic" in df.columns else np.zeros(len(df))

# Line 758 — `alert_types` feature pulls from the alert_type LABEL column (binary)
if "alert_type" in df.columns:
    alert_types = (df["alert_type"] == "attack").astype(int).values

# Lines 902–907 — the 6 "baseline" features assembled in this exact order:
features = np.column_stack([
    tactics, alert_types, hour, dow, protocols, services,   # ← dims 0..5 (baseline)
    src_bytes, dst_bytes, severity, src_port, dst_port,     # ← dims 6..10
    stages, device_types, techniques, campaigns,             # ← dims 11..14
    burst_1s, burst_10s, burst_60s, tail_sbytes, tail_dbytes,
    source_ids,
])
```

When the model is configured with `alert_feature_dim=6`, it consumes `features[:, :6]` — which is exactly the leaked pair plus 4 safe features.

### Stage B methodology

**Script:** `scripts/test_leakage_materiality.py`

**Strategy:** monkey-patch `AlertToGraphConverter._encode_alert_features()` to shuffle columns 0 (`tactics`) and 1 (`alert_types`) across rows with a fixed seed (42), preserving the marginal distribution but destroying the per-row label correspondence. Then re-run `benchmark.run_benchmark` on each frozen baseline config with the existing checkpoint. No retraining; no other code change.

**Configs tested:** 8 total — 6 v1.0 baselines (NSL-KDD, UNSW-NB15, TON-IoT, CICIDS2017, SQTK_SIEM, DARPA OpTC) + 2 v1.1 freezes (TON-IoT GMM+BIC, SQTK_SIEM pca=11).

**Raw output:** `experiments/results/leakage_materiality.csv` (8 rows), `experiments/results/leakage_audit_summary.csv` (aggregate), `experiments/results/leakage_audit_log.txt`.

**Verdict thresholds:**
- |Δ| < 0.02 → inert
- 0.02 ≤ |Δ| < 0.05 → mild
- |Δ| ≥ 0.05 → material

## Per-Dataset Implications

### NSL-KDD v1.0 — INVALIDATED
The +0.188 margin over Spectral(raw) was almost entirely leakage-driven. Clean ARI 0.093 falls well below all competing baselines. The "zero-shot wins on classic IDS" framing for this dataset cannot be supported by v1.0 evidence. Decision: per-dataset doc gets correction banner; freeze artifacts stay; v2.0 retrain queued.

### UNSW-NB15 v1.0 — INVALIDATED
The +0.210 margin over PCA+HDBSCAN was largely leakage. Clean ARI 0.289 < PCA+HDBSCAN 0.354. Same treatment as NSL-KDD.

### TON-IoT v1.0 — MIXED (superseded)
Δ=−0.073 is material but moderate. Irrelevant for decisions because v1.1 supersedes it.

### TON-IoT v1.1 (GMM+BIC) — GENUINE
The strongest validated result in the program. Δ=−0.011 is inert. Clean ARI 0.593 vs v1.0 clean 0.350 → +0.243 honest improvement (vs +0.181 reported). The GMM+BIC clustering displaces the leaked features' contribution by selecting clusters from the broader embedding geometry rather than concentrating on the two leaked dims. Decision: v1.1 freeze stands fully; doc banner notes the GENUINE status.

### CICIDS2017 v1.0 — GENUINE (honest loser)
Δ=+0.009 inert (slight improvement when leakage removed, probably noise). The model could not exploit the leak here because tactic is null/UNKNOWN for the 72.9% BENIGN class. The reported 0.177 ARI IS the honest baseline. The failure mode (BENIGN sub-structure not learnable) is real and will be the primary target of v2.0 retraining experiments.

### SQTK_SIEM v1.0 — INVALIDATED
Δ=−0.167 material. Clean ARI 0.188 vs PCA+HDBSCAN 0.382 → V3 loses by even more honestly than reported. Compound finding with the earlier 2026-05-23 cosine_sim correction.

### SQTK_SIEM v1.1 (pca=11) — INVALIDATED + REGRESSES
Cleanest case study of why the audit was worth running. Clean ARI 0.162 < v1.0 clean 0.188 → the pca=11 freeze is a REGRESSION under honest measurement. The "knife-edge optimum at pca=11" was a leakage interaction artifact. Frozen artifacts stay (immutability) but the recommendation cannot propagate. The v1.2 validity ledger marks this freeze INVALIDATED.

### DARPA OpTC v1.0 — CATASTROPHIC
Clean binary_ari = −0.020 (below random). The "near-perfect 0.999" was 100% leakage on a binary task where the binary label is fed in literally as `alert_types`. The trivially-perfect score this produced was a direct function of the feature being identical to the label. The `multidomain_v2_optc_finetuned` checkpoint shares the same input pipeline and likely shows similar collapse — flagged for v2.0 priority testing.

## Process Improvements (now in force)

1. **Pre-flight leakage check required** for every experiment subplan. For each input feature dimension the model will receive, document the source column and confirm it is NOT one of the evaluation label tracks (`label_col` or any `alt_label_cols` in `benchmark/datasets_real.yaml`). Verify by reading the actual feature-extraction code, not by trusting documentation.
2. **"Frozen" semantics clarified.** From v1.2 onward, "frozen" means reproducibility-frozen (the pipeline reproduces the recorded numbers), not honesty-validated. Honesty-validation is a separate dimension assessed via shuffle tests or clean-baseline comparison.
3. **CLAUDE.md and per-dataset baseline docs** must report both reported and clean ARIs when both are known. The reported number is fine for pipeline-internal comparisons; the clean number is required for external comparisons (papers, baselines, claims of zero-shot generalization).
4. **Lifecycle template update** (deferred to a separate small batch): add an "Input Feature Provenance" section to `docs/datasets/_template/audit.md` and a "Pre-Flight Label Leakage Check" section to `docs/experiments/_template/pre_flight.md`.

## Open Questions / Future Work

1. **v2.0 clean baseline retrain** (Part X.A.VII): retrain `network_v9_v3` with `tactics` and `alert_types` dropped from inputs. Likely candidate baseline is 4-dim (hour, dow, protocols, services) or 10-dim (4 safe + 6 safe new from the Exp 3 audit). Cost: ~4h GPU + ~3h benchmark + ~4h docs.
2. **Should `multidomain_v2_optc_finetuned` be re-tested under the shuffle protocol?** Likely yes — if its 0.897 ARI on UNSW-NB15 is similarly leakage-driven, the entire "supervised reference" line in CLAUDE.md needs revision. Cheap (~15 min).
