## Root Cause of V3 Underperformance

See `docs/datasets/ton_iot/investigation.md` for full analysis. Summary:

1. **HDBSCAN over-segmentation**: V3 predicts 55 clusters for 10 true classes. `min_cluster_size=5` (from NSL-KDD config) splits each ~1,000-row true class into multiple sub-clusters on TON_IoT's balanced data.
2. **K-Means privileged prior**: K-Means receives `n_clusters=10` — exactly the true class count. This is a form of privileged label information not available in a real deployment.
3. **AMI is competitive (0.705 vs 0.769)**: V3 captures cluster structure well; the failure is the HDBSCAN cluster count, not the HGNN representation. AMI penalises over-segmentation less than ARI.
4. **Graph structure reduction**: Missing `hostname`/`username` columns → no host/user entity nodes. Graph is IP-only. Impact unquantified but likely non-zero.
5. **Checkpoint trained on network IDS (not IoT)**: `network_v9_v3` learned on NSL-KDD/UNSW-style traffic; IoT protocol patterns may not transfer.

**Sweep finding:** Targeted mcs sweep improved eval ARI from 0.423 → 0.474 (mcs=300), but gap to K-Means (raw) remained 0.148 ARI.

## Known Limitations

1. **V3 loses on TON_IoT** — this is a genuine finding. The zero-shot claim holds for network IDS (NSL-KDD, UNSW-NB15) but not for IoT traffic with the current checkpoint.
2. **K-Means unfair advantage**: n_clusters=10 is set to the true class count. Fairness requires noting this in any paper comparison.
3. **Single IoT checkpoint**: `siem_supcon_v4` was trained on SQTK_SIEM, not TON_IoT. No TON_IoT-specific checkpoint exists; fine-tuning might recover the margin.
4. **Missing features**: `src_bytes`, `dst_bytes`, `service` absent → reduced feature dimensionality vs NSL-KDD/UNSW.
